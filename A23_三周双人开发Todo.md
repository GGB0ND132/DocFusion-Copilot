# A23 DocFusion Copilot 三周双人开发 Todo

适用前提：
- 当前时间按 2026-03-25 计算，截止时间为 2026-04-18。
- 团队仅 2 人。
- 当前仓库仍处于方案阶段，目标不是做“大而全”产品，而是做一个能稳定演示、能解释、能答辩的比赛版 MVP。

核心原则：
- 先打通“文档上传 -> 解析入库 -> Excel 模板回填 -> 来源追溯”闭环，再考虑扩展。
- 所有任务按 Must / Optional 区分，Must 为必须完成，Optional 为时间允许再做。
- 只保留与比赛评分直接相关的功能，避免在非关键功能上消耗时间。

## 1. 总目标

截止 4 月 18 日前，必须交付以下最小可演示闭环：

- [Must] 支持上传 docx、md、xlsx、txt 文档。
- [Must] 支持异步解析文档并展示任务状态。
- [Must] 支持抽取结构化事实并保存来源信息。
- [Must] 支持上传 Excel 模板并自动回填核心字段。
- [Must] 支持结果下载。
- [Must] 支持点击或查看填充值对应的来源证据。
- [Must] 支持最基本的异常处理和空值兜底。
- [Must] 支持一套完整 Demo 流程和答辩材料。

可以作为加分项但不是必做：

- [Optional] Word 模板回填。
- [Optional] 自然语言 Agent 指令交互。
- [Optional] OCR 兜底。
- [Optional] 低置信度高亮。
- [Optional] 统计看板或漂亮的大屏。
- [Optional] Docker Compose 一键部署。

## 8. 风险清单

### 高风险

- [Must] 模板理解不稳定，导致字段映射错误。
- [Must] 数值单位换算错误，导致填表看起来很离谱。
- [Must] 事实抽取没有证据链，答辩时解释不清。
- [Must] 两个人同时开发但没有固定接口，后期联调爆炸。

### 对策

- [Must] 先固定一类模板和一类样例文档，优先把单场景做稳。
- [Must] 所有数值字段都走规则校验。
- [Must] 每条回填结果必须能追溯到原文。
- [Must] 每晚同步一次接口字段和状态码。

## 11. 一句话结论

你们现在不是要做“完整产品”，而是要在三周内做出一个评分点明确、闭环完整、演示稳定、答辩能讲透的比赛版系统。所有 Optional 功能都不能影响 Must 功能交付。
